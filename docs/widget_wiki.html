Windows Platform Reference

Windows Surface & Agent Extensibility

A consolidated reference covering the user-facing surfaces that Windows exposes for third-party content and assistance: Widgets on the Windows 11 Widgets Board, legacy Live Tiles on the Start screen, the Microsoft Agent / Clippy assistant lineage, Windows Terminal as a host for command-line agents, the Windows NT kernel and COM activation model that underpins them all, and finally Windows Clippy MCP — the project shipping in this repository that turns the desktop itself into a tool surface for AI agents.

How the surfaces fit together

Every chapter in this wiki describes a distinct extensibility surface. They differ in lifetime, host process, packaging story, and primary content format, but they share the same activation foundations: COM, the Windows kernel, MSIX, and (more recently) AppExtension contracts.

+--------------------------+ | Windows User Session | +-----+----------+---------+ | | +----------------+ +----------------+ | | +-------v-------+ +---------------+ +--------v--------+ | Widgets Host | | StartMenu / | | Console / Term | | (Win11 Board) | | Live Tiles | | (conhost, | | dashost.exe | | (Win8/8.1/10) | | WindowsTerm) | +-------+-------+ +-------+-------+ +--------+--------+ | | | | OOP COM | Toast/Tile XML | ConPTY | (CLSID + AppExt) | + WNS push | + UIA v v v +-------+-------+ +-------+-------+ +--------+--------+ | Provider EXE | | Tile XML + | | Shell / agent | | (this repo's | | Notif Mgr | | process (e.g. | | tutorial) | | | | Clippy MCP) | +-------+-------+ +---------------+ +--------+--------+ | | +-----------+ Win32 / NT syscalls +----+ v v +-------+---------------------------+--------+ | ntoskrnl.exe — scheduler, object manager, | | ALPC, registry, MSIX container, security | +--------------------------------------------+
  • Widgets are short-lived Adaptive Card surfaces hosted out of process by the Widgets Board. Activation is COM-based via a CLSID registered through MSIX.
  • Live Tiles are XML notifications rendered by the Start menu. They are deprecated on Windows 11 but still ship in maintenance mode and remain instructive as a contrast.
  • Clippy / Microsoft Agent is the historical thread of in-process character assistants — from Office Assistant (1997) through MS Agent COM, to today's Copilot.
  • Windows Terminal is the modern multiplexed console host. Agents like Copilot CLI run as child processes attached via ConPTY.
  • Kernel chapter covers the NT primitives every surface above leans on: object manager, ALPC, registry, security, and the user/kernel boundary.
  • Windows Clippy MCP is this repository — a Python MCP server exposing 49+ tools that drive the desktop on behalf of an LLM agent.
Chapter 1

Windows 11 Widgets — IWidgetProvider in C#

End-to-end walkthrough: build a provider, define Adaptive Card templates, package as MSIX, register the COM CLSID, and ship two widgets (Weather and Counter).

Overview

The widget host calls methods on IWidgetProvider to request data that defines a widget or to let the provider respond to user actions. This example defines two widgets:

  • Weather Widget — a mock weather card illustrating Adaptive Cards formatting options, including background image, columns, and text blocks.
  • Counting Widget — demonstrates user actions and custom widget state by maintaining a counter that increments when the user clicks a button.

Sample code is adapted from the Windows App SDK Widgets Sample. To implement a widget provider in C++/WinRT, see the separate C++/WinRT walkthrough.

Prerequisites

  • Developer mode must be enabled on your device.
  • Visual Studio 2026 or later with the WinUI application development workload.
  • Add the optional component C++ (v143) from the WinUI workload.

  3. Set the target .NET version to 8.0.

Prepare to Track Enabled Widgets

Declare a helper structure to store the ID, name, data, and active state for each pinned widget. Add this inside the ExampleWidgetProvider namespace, outside the WidgetProvider class:

C#WidgetProvider.cs
public class CompactWidgetInfo
{
    public string? widgetId { get; set; }
    public string? widgetName { get; set; }
    public int customState = 0;
    public bool isActive = false;

Inside the WidgetProvider class, add the running widgets dictionary:

public static Dictionary<string, CompactWidgetInfo> RunningWidgets
    = new Dictionary<string, CompactWidgetInfo>();

Implement the IWidgetProvider Methods

NOTE: Objects passed into IWidgetProvider callbacks are only valid within the callback. Do not store references to them — behavior outside the callback is undefined.

CreateWidget

Called when the user pins one of your app's widgets. Gets the ID and name, adds a CompactWidgetInfo entry, then calls UpdateWidget.

C#WidgetProvider.cs
public void CreateWidget(WidgetContext widgetContext)
{
    var widgetId   = widgetContext.Id;
    var widgetName = widgetContext.DefinitionId;
    CompactWidgetInfo runningWidgetInfo = new CompactWidgetInfo()
        { widgetId = widgetId, widgetName = widgetName };
    RunningWidgets[widgetId] = runningWidgetInfo;
    UpdateWidget(runningWidgetInfo);
}

DeleteWidget

Called when the user unpins a widget. Removes it from the running widgets map. If the map becomes empty, sets an event to allow the app to exit gracefully.

C#WidgetProvider.cs
public void DeleteWidget(string widgetId, string customState)
{
    RunningWidgets.Remove(widgetId);
    if (RunningWidgets.Count == 0)
        emptyWidgetListEvent.Set();
}

static ManualResetEvent emptyWidgetListEvent = new ManualResetEvent(false);
public static ManualResetEvent GetEmptyWidgetListEvent() => emptyWidgetListEvent;

OnActionInvoked

Called when the user interacts with an action in the widget template. Check the Verb property to determine which action was triggered. For the counter widget, "inc" increments the custom state counter.

C#WidgetProvider.cs
public void OnActionInvoked(WidgetActionInvokedArgs actionInvokedArgs)
{
    var verb = actionInvokedArgs.Verb;
    if (verb == "inc")
    {
        var widgetId = actionInvokedArgs.WidgetContext.Id;
        var data     = actionInvokedArgs.Data;
        if (RunningWidgets.ContainsKey(widgetId))
        {
            var localWidgetInfo = RunningWidgets[widgetId];
            localWidgetInfo.customState++;
            UpdateWidget(localWidgetInfo);
        }
    }
}

OnWidgetContextChanged

In the current release, called when the user resizes a pinned widget. You can return a different template per size, use conditional rendering in the template JSON, or use this callback for telemetry only.

C#WidgetProvider.cs
public void OnWidgetContextChanged(WidgetContextChangedArgs contextChangedArgs)
{
    var widgetContext = contextChangedArgs.WidgetContext;
    var widgetId      = widgetContext.Id;
    var widgetSize    = widgetContext.Size;
    if (RunningWidgets.ContainsKey(widgetId))
        UpdateWidget(RunningWidgets[widgetId]);
}

Activate and Deactivate

Activate notifies the provider that the widget host is actively displaying the widget and wants fresh data. Deactivate signals the opposite. Providers can send updates at any time, but balance freshness against battery life. Both are per-widget callbacks.

C#WidgetProvider.cs
public void Activate(WidgetContext widgetContext)
{
    var widgetId = widgetContext.Id;
    if (RunningWidgets.ContainsKey(widgetId))
    {
        var localWidgetInfo = RunningWidgets[widgetId];
        localWidgetInfo.isActive = true;
        UpdateWidget(localWidgetInfo);
    }
}

public void Deactivate(string widgetId)
{
    if (RunningWidgets.ContainsKey(widgetId))
        RunningWidgets[widgetId].isActive = false;
}

Update a Widget

The UpdateWidget helper constructs WidgetUpdateRequestOptions with the appropriate template JSON, data JSON, and custom state, then calls WidgetManager.GetDefault().UpdateWidget().

C#WidgetProvider.cs
void UpdateWidget(CompactWidgetInfo localWidgetInfo)
{
    WidgetUpdateRequestOptions updateOptions
        = new WidgetUpdateRequestOptions(localWidgetInfo.widgetId);

    string? templateJson = null;
    if      (localWidgetInfo.widgetName == "Weather_Widget")
        templateJson = weatherWidgetTemplate;
    else if (localWidgetInfo.widgetName == "Counting_Widget")
        templateJson = countWidgetTemplate;

    string? dataJson = null;
    if      (localWidgetInfo.widgetName == "Weather_Widget")
        dataJson = "{}";
    else if (localWidgetInfo.widgetName == "Counting_Widget")
        dataJson = "{ \"count\": " + localWidgetInfo.customState.ToString() + " }";

    updateOptions.Template    = templateJson;
    updateOptions.Data        = dataJson;
    updateOptions.CustomState = localWidgetInfo.customState.ToString();
    WidgetManager.GetDefault().UpdateWidget(updateOptions);
}

Initialize the List of Enabled Widgets on Startup

In the constructor, query WidgetManager for any already-running widgets to recover state after a restart or crash.

C#WidgetProvider.cs — constructor
public WidgetProvider()
{
    var runningWidgets = WidgetManager.GetDefault().GetWidgetInfos();
    foreach (var widgetInfo in runningWidgets)
    {
        var widgetContext = widgetInfo.WidgetContext;
        var widgetId      = widgetContext.Id;
        var widgetName    = widgetContext.DefinitionId;
        var customState   = widgetInfo.CustomState;
        if (!RunningWidgets.ContainsKey(widgetId))
        {
            CompactWidgetInfo info = new()
                { widgetId = widgetId, widgetName = widgetName };
            try { info.customState = Convert.ToInt32(customState); }
            catch { }
            RunningWidgets[widgetId] = info;
        }
    }
}

Implement a Class Factory

CoRegisterClassObject requires an IClassFactory implementation. Add a new class file FactoryHelper.cs and replace its contents:

C#FactoryHelper.cs
using Microsoft.Windows.Widgets.Providers;
using System.Runtime.InteropServices;
using WinRT;

namespace COM
{
    static class Guids
    {
        public const string IClassFactory = "00000001-0000-0000-C000-000000000046";
        public const string IUnknown      = "00000000-0000-0000-C000-000000000046";
    }

    [ComImport, ComVisible(false),
     InterfaceType(ComInterfaceType.InterfaceIsIUnknown),
     Guid(COM.Guids.IClassFactory)]
    internal interface IClassFactory
    {
        [PreserveSig]
        int CreateInstance(IntPtr pUnkOuter, ref Guid riid, out IntPtr ppvObject);
        [PreserveSig]
        int LockServer(bool fLock);
    }

    [ComVisible(true)]
    class WidgetProviderFactory<T> : IClassFactory
        where T : IWidgetProvider, new()
    {
        public int CreateInstance(IntPtr pUnkOuter, ref Guid riid, out IntPtr ppvObject)
        {
            ppvObject = IntPtr.Zero;
            if (pUnkOuter != IntPtr.Zero)
                Marshal.ThrowExceptionForHR(CLASS_E_NOAGGREGATION);
            if (riid == typeof(T).GUID || riid == Guid.Parse(COM.Guids.IUnknown))
                ppvObject = MarshalInspectable<IWidgetProvider>.FromManaged(new T());
            else
                Marshal.ThrowExceptionForHR(E_NOINTERFACE);
            return 0;
        }
        int IClassFactory.LockServer(bool fLock) => 0;

        private const int CLASS_E_NOAGGREGATION = -2147221232;
        private const int E_NOINTERFACE        = -2147467262;
    }
}

Register the Widget Provider Class Object with OLE

Replace Program.cs with the following. Replace the placeholder GUID with the one you generated.

C#Program.cs
using System.Runtime.InteropServices;
using ComTypes = System.Runtime.InteropServices.ComTypes;
using Microsoft.Windows.Widgets;
using ExampleWidgetProvider;
using COM;

[DllImport("kernel32.dll")]
static extern IntPtr GetConsoleWindow();

[DllImport("ole32.dll")]
static extern int CoRegisterClassObject(
    [MarshalAs(UnmanagedType.LPStruct)] Guid   rclsid,
    [MarshalAs(UnmanagedType.IUnknown)] object pUnk,
    uint dwClsContext, uint flags, out uint lpdwRegister);

[DllImport("ole32.dll")]
static extern int CoRevokeClassObject(uint dwRegister);

Console.WriteLine("Registering Widget Provider");
uint cookie;
Guid CLSID_Factory = Guid.Parse("XXXXXXXX-XXXX-XXXX-XXXX-XXXXXXXXXXXX");
CoRegisterClassObject(CLSID_Factory, new WidgetProviderFactory<WidgetProvider>(),
    0x4, 0x1, out cookie);

if (GetConsoleWindow() != IntPtr.Zero)
{
    Console.WriteLine("Registered successfully. Press ENTER to exit.");
    Console.ReadLine();
}
else
{
    using (var evt = WidgetProvider.GetEmptyWidgetListEvent())
        evt.WaitOne();
    CoRevokeClassObject(cookie);
}

Implementing Widget Customization

Starting with Windows App SDK 1.4, widgets can support user customization. A Customize widget option appears in the ellipsis menu above Unpin widget.

KNOWN BUG: There is a known bug with the Windows Widget Board for widgets built using the Windows App SDK that causes the ellipsis menu to become unresponsive after the customization card is shown.

Customization flow summary:

  1. The widget host calls the regular template/data payloads for normal operation.
  2. User clicks Customize widget in the ellipsis menu.
  3. The widget raises OnCustomizationRequested on the provider.
  4. The provider sets an internal flag and sends the customization UI template instead of the regular template.
  5. OnActionInvoked events fire as the user interacts with the customization UI.
  6. On the app-defined "exit customization" action, the provider clears the flag and resumes normal templates. If the user exits without clicking the exit action, IWidgetProviderAnalytics.OnAnalyticsInfoReported fires with interactionKind: "exitCustomization".
  7. The provider persists customization options to disk or cloud.
WARNING: Widget customization requires Windows App SDK 1.4 or later. Update the NuGet package reference before implementing.

Respond to Customization Actions

Extend OnActionInvoked to handle the "reset" and "exitCustomization" verbs:

C#WidgetProvider.cs — OnActionInvoked (complete)
public void OnActionInvoked(WidgetActionInvokedArgs actionInvokedArgs)
{
    var verb     = actionInvokedArgs.Verb;
    var widgetId = actionInvokedArgs.WidgetContext.Id;

    if (!RunningWidgets.ContainsKey(widgetId)) return;
    var info = RunningWidgets[widgetId];

    if (verb == "inc")
    {
        info.customState++;
        UpdateWidget(info);
    }
    else if (verb == "reset")
    {
        info.customState    = 0;
        info.inCustomization = false;
        UpdateWidget(info);
    }
    else if (verb == "exitCustomization")
    {
        info.inCustomization = false;
        UpdateWidget(info);
    }
}

After deploying, clicking Customize widget in the ellipsis menu displays the customization template. Reset counter resets the counter to 0. Exit customization returns the widget to its normal view.

Live Tiles — the Start screen surface (legacy)

The XML-driven notification system that powered Windows 8, 8.1, and 10 Start tiles. Effectively superseded by Widgets on Windows 11, but still ships and remains the clearest contrast point for understanding what Widgets replaced.

Overview

A Live Tile is a rectangular surface pinned to the Windows Start screen that an app can periodically update with content — text, images, badges, or peeking animations. Unlike a Widget, a Tile is owned by the app's identity and is updated by delivering an XML payload to the Notification Platform; the platform itself renders the tile based on a fixed catalogue of templates.

Status: maintenance Introduced: Windows 8 (2012) Peak: Windows 10 (2015) Win11 Start: live updates removed API surface: Windows.UI.Notifications

History & Status

Tile Anatomy

A tile is composed of three logically separate notification streams:

  • Tile content — the foreground text/image surface, defined by a <tile> XML payload sent through TileUpdateManager.
  • Badge — a small numeric or glyph overlay (mail count, alarm icon), sent through BadgeUpdateManager.
  • Toast — transient banner notifications using a separate <toast> XML schema. Toasts and tiles share the WNS pipeline but render independently.

Tiles can be the app's primary tile (one per app, derived from the manifest), or one of many secondary tiles created at runtime via SecondaryTile.

Templates & XML

Modern Live Tiles use the Adaptive Tile schema. A single payload can contain multiple <binding> elements — one per target size — and Windows picks the binding that matches the user's current tile size.

XMLAdaptive tile payload (medium + wide)
<tile>
  <visual>
    <binding template="TileMedium">
      <text hint-style="caption">Mail</text>
      <text hint-style="captionSubtle">3 unread</text>
      <image src="ms-appx:///Assets/inbox.png" placement="peek" />
    </binding>
    <binding template="TileWide">
      <group>
        <subgroup>
          <text hint-style="subtitle">Inbox</text>
          <text hint-style="captionSubtle">3 unread · 12 today</text>
        </subgroup>
      </group>
    </binding>
  </visual>
</tile>
NOTE: The legacy fixed-template names (e.g. TileSquare150x150Image, TileWide310x150ImageAndText01) still validate but should not be used in new payloads. Adaptive bindings give the same expressive range with a clean grammar.

Secondary Tiles

A secondary tile pins a deep-linked area of the app — a chat conversation, a stock symbol, a folder — to the Start screen as its own tile. Created at runtime, each tile has an independent updater stream and its own badge.

ACS / ACD Files

Characters were packaged as .acs files (single-file character with all animations) or .acd + per-animation files for incremental download. The format embedded:

  • Character header (name, GUID, default colors, balloon font).
  • An animation table — each animation a sequence of frames with inter-frame deltas, mouth-shape codes for lip-sync, and branch logic.
  • An optional voice profile (TTS engine ID and parameters).
  • State definitions (Idle, Speaking, Listening, Hide) that the runtime mapped onto user-visible behavior.

The acdgen.exe compiler shipped in the Microsoft Agent Character Editor took a folder of BMP frames and a script and emitted a runnable .acs.

Windows Terminal — the modern console host

The MIT-licensed multiplexed console host. Where conhost and cmd.exe used to be the only game in town, Terminal hosts arbitrary shells through ConPTY and is the natural runtime surface for command-line agents like GitHub Copilot CLI.

Overview

Windows Terminal (executable WindowsTerminal.exe, package Microsoft.WindowsTerminal) is a tabbed, GPU-accelerated terminal emulator first released in 2019 as an open-source replacement for the legacy console host. It does not implement a shell itself; it hosts shells (PowerShell, cmd, WSL, ssh, Azure Cloud Shell, etc.) as child processes attached via the ConPTY API.

Status: actively developed Repo: microsoft/terminal (MIT) Default on Win11 22H2+ Settings: JSON

YearMilestone

2018Windows 10 1809 introduces the ConPTY API (CreatePseudoConsole) — the OS now exposes a real pseudo-TTY interface, decoupling shells from conhost's renderer.
2019Windows Terminal 1.0 ships from microsoft/terminal. DirectWrite-rendered, JSON-configured, tabbed.

ConPTY

CreatePseudoConsole creates a pair of pipes that look like a terminal to the child process — VT input flows in, VT-encoded output flows back. The renderer (Windows Terminal, an SSH server, a remote Visual Studio host) consumes the VT stream however it pleases. This is the same architectural shape Unix has always had with openpty; Windows shipped the equivalent decades later.

+-------------+ in pipe (VT) +-------------+ | Renderer | ------------------> | Child shell| | (Terminal, | | (pwsh, | | ssh, etc.) | <------------------ | bash, ssh) | +-------------+ out pipe (VT) +-------------+ | | | CreatePseudoConsole(hpc, ...) | +----------------------------------+ | v ntoskrnl ALPC + I/O

NT Kernel & COM Activation

The OS-level mechanics every surface above depends on. Widgets, tiles, agents, and the MCP server all reduce eventually to processes activated via COM, scheduled by the NT kernel, and mediated by the object manager and ALPC.

Overview

Windows is built on the NT kernel (ntoskrnl.exe), a hybrid microkernel/monolithic design originally written by Dave Cutler's team starting in 1989. Every user-mode process — including dashost.exe (the widget host), Windows Terminal, and Python running this MCP server — sits on top of the same primitives: the object manager, the ALPC IPC system, the registry, the security reference monitor, the I/O manager, and the Win32 subsystem.

Architecture

User mode +----------------------------------------------------------------+ | Win32 apps UWP / WinAppSDK .NET / CoreCLR Python ... | +-------+----------+----------------+----------------+-----------+ | | | | v v v v +-------+----------+----------------+----------------+-----------+ | kernel32 / user32 / advapi32 | ntdll.dll | | combase / ole32 / rpcrt4 | (syscall stubs) | +-------+--------------------+-------+----------+----------------+ | csrss | wininit | lsass | (Win32 subsys) | (boot) | (security) v v v Kernel mode (ring 0) +----------------------------------------------------------------+ | ntoskrnl.exe | | +-- Executive (Object Manager, IO, Mem, Cache, Config) | | +-- Kernel (scheduler, syscalls, ALPC) | | +-- win32k.sys (window manager, GDI) | | +-- HAL | +----------------------------------------------------------------+

Object Manager

Every kernel-managed resource — files, registry keys, threads, processes, mutexes, sections, ALPC ports, devices — is an object in a single global namespace under \. User-mode code receives handles to those objects; the object manager enforces access checks via the security descriptor on each object.

You can browse the live namespace with Sysinternals WinObj. Of particular interest for surfaces in this wiki:

  • \RPC Control\ — ALPC ports used by COM and RPC services.
  • \BaseNamedObjects\ — named mutexes / events used by user-mode apps.
  • \Sessions\<n>\ — per-session subtrees for terminal-services-style isolation.
  • \Device\ — device objects exposed by drivers.

User / Kernel Boundary

User-mode code never calls kernel functions directly. ntdll.dll contains a stub for every system service (e.g. NtCreateFile) that loads the service number into eax and issues syscall (x64) or sysenter (x86). The CPU traps to ring 0, the kernel dispatches by service number through KiSystemServiceTable, and returns with a status code.

Public Win32 APIs (kernel32!CreateFile) are thin wrappers over these Nt* entry points. Modern Windows also exposes the WoW64 transition for 32-bit processes on 64-bit kernels, and Pico processes (the WSL1 mechanism) which let a separate provider drive system calls.

COM Activation

The Component Object Model is how Windows surfaces — including Widgets — bind a stable identifier (a CLSID) to a piece of executable code. There are four activation flavors:

FlavorServer typeLoaded into
In-processDLLThe caller's process.
Local out-of-processEXEA separate server EXE on the same machine. Used by IWidgetProvider.
SurrogateDLL hosted by dllhost.exeA generic surrogate that loads the DLL out-of-proc for isolation.
Remote (DCOM)EXE on another machineLargely deprecated in modern apps.

Registration historically lived under HKCR\CLSID\{guid}, with subkeys InProcServer32 / LocalServer32 pointing at the binary. Packaged apps register declaratively — the manifest's com:Extension element creates the same registry entries scoped to the MSIX package's identity, removing them when the package is uninstalled.

XMLPackage.appxmanifest — out-of-proc COM server (Widget provider)
<com:Extension Category="windows.comServer">
  <com:ComServer>
    <com:ExeServer
        Executable="WidgetProviderApp\WidgetProviderApp.exe"
        DisplayName="Widget Provider Sample"
        Arguments="-RegisterProcessAsComServer">
      <com:Class Id="<your-CLSID-here>" DisplayName="WidgetProvider" />
    </com:ExeServer>
  </com:ComServer>
</com:Extension>

MSIX & AppContainer

MSIX is the modern Windows packaging format. A .msix is a signed ZIP of the app payload plus an AppxManifest.xml. At install time the package is mounted into a per-app virtual file system root and a per-app virtual registry hive, both torn down on uninstall.

Packaged processes can run inside an AppContainer — a sandbox that limits access to the file system, registry, and named objects according to declared capabilities. Widgets typically run in runFullTrust mode for compatibility with the existing C# console-app pattern in Chapter 1, but can also run inside a partial-trust container if all dependencies cooperate.

AppExtension Contract

windows.appExtension is a manifest-declared contract that lets one packaged app advertise an extension and another packaged app discover it at runtime through AppExtensionCatalog.Open. This is the activation grammar that the Widget host uses to find your provider and that Windows Terminal's fragment search fits into.

  • The provider declares uap3:Extension Category="windows.appExtension" with a Name matching the host's contract ID.
  • The host opens the catalogue for that contract ID, enumerates packages, and reads each provider's Properties XML to bootstrap activation.
  • Discovery is real-time — install a new provider package and the host fires PackageInstalled.

What is MCP?

The Model Context Protocol is an open JSON-RPC 2.0 spec for letting LLM clients discover and call tools, read resources, and execute prompts exposed by a separate server process. An MCP server is just a process that speaks JSON-RPC over stdio (or SSE / WebSocket) and advertises a manifest of capabilities.

Adding Tools

The tool authoring shape is one decorator over one function:

Pythonmain.py — adding a new tool
@mcp.tool(name='NewTool-Name', description='Clear description of functionality')
def new_tool(param1: str, param2: int = 0) -> str:
    try:
        # Use the desktop wrapper or pyautogui directly
        result = do_work(param1, param2)
        return f'Success: {result}'
    except Exception as e:
        return f'Error: {e}'

Conventions enforced by the contributor guide:

  • Tool name is <Function>-Tool.
  • Always wrap the body in try/except and return a string. Never raise — the LLM client surfaces strings cleanly and re-raised exceptions break the JSON-RPC frame.
  • Update the tool count in README.md and add an entry to the family table.
  • Validate syntax (uv run python -m py_compile main.py) and exercise the tool on a real Windows host before committing.

Security Notes

KNOWN BUG: Treat the MCP process as you would any agent that has full keyboard, mouse, registry, and PowerShell access to the user's session. There is no sandbox. Run only locally, against models you trust, and never expose the stdio transport to a network listener without an authentication wrapper.
  • Powershell-Tool runs arbitrary commands as the current user. The same caveat applies to Registry-Tool writes (currently read-only) and Process-Tool kill.
  • Connect-MGGraph-Tool uses interactive browser auth; tokens live in the user's profile cache. Treat them as scoped credentials.
  • UI Automation can read passwords typed into other windows. Be deliberate about State-Tool on screens with sensitive content.
